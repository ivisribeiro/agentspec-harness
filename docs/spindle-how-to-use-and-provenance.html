It covers how to use the agents, commands, and flows; a capability comparison against AgentSpec, ECC, and OpenSpec; and an honest, adversarially-reviewed provenance breakdown of what was forked, ported, borrowed, and originally built.

Harness-native spec-driven dataops

Spindle

Spindle is a Claude Code plugin that turns a spec-driven-development workflow into something actually testable. A deterministic CLI named spin owns every "done" decision and returns it as a process exit code. The slash commands are the only layer that ever runs a model.

The hard seam

The model proposes; spin disposes. spin is a TypeScript CLI that never calls a model — a CI grep-guard fails the build if any network egress appears in src/. Completion is recorded only by spin complete --handoff (which validates the worker's JSON against a Zod schema, then marks done). Phases clear only through spin gate <id>, a pure function reading the filesystem and run-state — never the conversation. An LLM cannot self-mark itself complete.

Part A

How to use Spindle

Agents are dispatched by slash commands. Workers author artifacts and write a JSON handoff sidecar; commands drive the spin spine — routing the model, validating the handoff, running gates, and branching on the exit code. Workers never call spin complete / spin gate themselves.

·What runs, and who decides

Two roles, kept strictly apart. Slash commands (in your Claude session) are the only place a model executes — they dispatch agents via the Task tool on a routed model tier. The spin CLI is deterministic shell plumbing: it orders artifacts, validates handoffs, runs gates, and exits with a code the command branches on. A gate is the sole arbiter of a phase — never a worker's self-reported checkbox.

A4Command surface & the exit-code ABI

Every command branches strictly on spin's four-value exit code. spin is deterministic and never runs a model; the model only runs inside the slash command.

0
pass
Gate cleared / handoff valid. Proceed.
1
blocked
Gate blocked or handoff invalid. STOP — do not advance, never fake completion.
2
usage
Bad invocation / argument error.
3
internal
CLI internal error.

Completion is recorded only by spin complete <id> --handoff, which validates the worker JSON against its Zod schema (G_HANDOFF) and only then marks the artifact done — no skippable validate-then-mark seam. Phases clear only via spin gate <id>. Every retry loop is bounded by spin retry <id> --inc then --ok; hitting the config-driven ceiling exits 1 = STOP.

Named gates · spin gate <id>
  • G_DEFINE · Why/What + ≥3 AC-n
  • G_DESIGN · manifest + valid handoff
  • G_BUILD · files exist, criteria-diff empty, evidence present
  • G_SHIP · all criteria passed, drift reconciled
  • G_KB_STRUCTURE · G_KB_COVERAGE
  • G_REVIEW_BLOCK · any surviving critical → block
  • G_AUDIT · G_OPS_CONFIG · G_PLAN
  • G_ROUTER_COVERAGE · G_HANDOFF
Deterministic verbs · spin <verb>
  • init --schema · start a run (sdd/kb/brownfield/fact-check)
  • state · status · next · order
  • route <kind> · tier + model resolution
  • tier · T0/T1/T2 orchestration axis
  • complete --handoff · retry --inc/--ok
  • validate · handoff-check · diff-criteria
  • spec-drift · config-drift · reconcile

Authenticity & provenance

The honest part. Every claim below is the adversarially-reviewed version — contested claims are stated in their corrected form. The LICENSE and file headers attribute all three upstreams accurately; the provenance is honest, not aspirational.

C1The honest share

Spindle is roughly one-third genuinely original engineering and two-thirds assembled-or-forked content. By file count it is mostly inherited AgentSpec; by engineering novelty and product thesis the ratio inverts — the original third is the entire reason the project exists.

Forked · AgentSpec ~65%

By file count. ECC's footprint is conceptual only — no ECC file was copied, so it does not appear on the bar. By engineering novelty the proportions invert: the original deterministic spine is the point, the OpenSpec port is its load-bearing foundation, and the AgentSpec fork is the payload it orchestrates.

C2Original to Spindle

OriginalSpindle · the harness spine + KB-authoring machinery
  • The spin CLI and its uniform 4-value exit-code ABI on a spine a CI guard proves never calls a model. correctedThe bare idea of distinct exit codes is not novel — AgentSpec's judge.py already uses 0/1/2/3/4, and OpenSpec's CLI has a binary 0/1 contract. Spindle's actual contribution is the combination: a uniform ABI across the whole CLI, plus the CI-tested no-model guard wrapping a gate registry. AgentSpec's codes live inside a single model-calling helper, not a no-model spine.
  • The named gate engine (src/core/gates/, 11 files) — pure (GateContext)⇒GateResult functions reading filesystem + run-state only. Includes the phantom-criterion check (block a passed AC that DEFINE never declared, via both-ways criteria-diff) and the verified_by evidence check (a passed criterion citing a path must point at a file that exists). No counterpart in OpenSpec or AgentSpec.
  • The crash-safe run-state ledger .spindle/run.json — CLI-written-only, atomic temp+rename, Zod-validated on every read. The keystone that makes gates idempotent and re-runnable after a crash. grep: 0 hits upstream
  • spec-drift as a first-class typed loop (corrected_spec/correction/reconciled), born from Spindle's own pix-brcode dogfood where a factually-wrong CRC criterion passed every gate. A different concept from OpenSpec's requirement-level deltas.
  • The 10 Zod handoff schemas + the atomic validate-then-mark seam inside spin complete --handoff — the non-skippable boundary that prevents an LLM from self-marking done.
  • The model-route resolver code (policy.ts + tiers.ts). correctedThe three-tier model-selection doctrine is borrowed from ECC. But two principles are Spindle/Ivis originals, not ECC lineage: the enforcement rules (verifier outranks-or-equals generator on critical gates; critical kinds never downgrade) appear nowhere in ECC, and the T0/T1/T2 orchestration-tier axis is the user's own cost-discipline lesson. All resolver code is original — no upstream file copied.
  • The no-model CI guard (test/e2e/guard.test.ts) — greps all of src/ for fetch( / api.anthropic.com / claude -p / @anthropic / openai and fails the build. Turns the deterministic-spine invariant into a verifiable property.
  • The 13 typed harness workers and the KB-authoring lane's gate machinery (kb-gates.ts). correctedThe actual 13 are: 4 workflow *-worker, the single challenger adversary (one agent, opus-routed), arch-worker + security-worker, kb-concept-worker, migrate dbt/spark/equivalence (3), and factcheck extract/verify (2). There is no separate "auditor-worker" agent — auditing is the /audit command dispatching the review workers.

Honest limit, on the record: gates certify structure and identity, not truth. They prove a file exists, a criterion is passed, a verifier is cited, criteria sets are consistent — they cannot prove a CRC value is correct or a prose claim is true.

The honest read

By file count, Spindle is ~60–70% forked AgentSpec content (52 specialist agents + 333 KB files + visual-explainer + most commands), ~10% ported OpenSpec core, and ~25–30% original Spindle code + new workers and skills. The 52 agents and 333 KB files that make Spindle look large are AgentSpec verbatim, and the HTML/visual feature is inherited two levels deep.

By engineering novelty and by what is the actual product thesis, the ratio inverts. The original deterministic spine — the spin CLI, the gate engine, the run-state ledger, spec-drift, the handoff schemas, the model-route resolver, and the no-model CI guard — is a thin but high-leverage slice: maybe 25–35% of the meaningful engineering and close to 100% of what is novel. It is what makes "an LLM SDD workflow that is actually testable" a real claim rather than a slogan.

Spindle did not write the specialists or the knowledge base — it wrote the testable, no-model harness that makes an LLM run them under gates, and it ported its graph foundation from OpenSpec.
