# The measured harness

Spindle's first three layers PREDICT and BLOCK: model routing predicts the cheapest
tier that can do a task; typed gates block work that isn't really done. Neither, on its
own, ever **measures** whether the prediction or the block was right. The measurement
layer closes that loop — and it is what makes Spindle its own product rather than another
spec-driven wrapper: it is the harness that records its own trajectory, proves its own
gates still fire, and accounts for every token it was told it spent.

Crucially, measurement lives entirely on the **deterministic, offline side of the
seam**. It is more pure functions over `.spindle/` — set-diffs, aggregation, replay. The
CLI never infers, never calls a model, never reaches the network. The model-free guard
(`scripts/guard-no-model-calls.js`) is extended to forbid tokenizers and pricing tables
in `src/` precisely so this layer cannot quietly grow model-awareness.

## The doctrine in one line

> Routing is a **prediction**. Gates are a **hypothesis**. The run-ledger is the
> **evidence**. The eval corpus is the **regression test on the hypothesis**. Budget
> reconciliation is the **accounting**.

## The three measurement surfaces

### 1. The run-ledger — `spin trace`

`run.json` carries an append-only `events[]` ledger: the run's trajectory
(`complete` / `gate` / `retry`) with timestamps, distinct from the current-state maps.
It is a pure superset written only at the existing CLI mutation points, so crash-safety
and idempotency hold — an unchanged gate re-run appends nothing. `spin trace` reads it
and prints the timeline plus a tier histogram and summed reported tokens.

### 2. The eval corpus — `spin eval`

A corpus of recorded fixtures (gate + inputs + expected verdict) replayed through the
**real** gate functions. A code change that makes a gate stop blocking what it used to
block is a regression the corpus catches. It runs in CI over the bundled `dist`, in
seconds, with no model and no network. `spin eval --strict` is fail-closed on incomplete
gate coverage — and the corpus now covers **all 11 registry gates** with both a pass and a
block case (the arg-file gates carry a sidecar; the state-coupled gates — G_DEFINE/DESIGN/
BUILD/SHIP and the KB gates — carry a real `.spindle/` tree, generated by
`scripts/gen-eval-fixtures.mjs`), so `npm run eval` runs `--strict` and a future gate added
without fixtures fails the build. This is a regression test on the harness's own judgment.

### 3. Budget reconciliation — `spin budget`

Workers report their own usage (`{tier, model?, tokens_in?, tokens_out?}`) onto the
handoff sidecar; `spin complete` records it opaquely into the ledger. `spin budget` sums
that reported spend per tier and, given an optional `--max-tokens` ceiling, flags
`over_budget`.

**Accounting, not enforcement.** Two honest limits, stated on the record:

- The CLI **cannot independently verify** a self-reported token count. If a worker
  under-reports or omits its `usage`, the total is wrong and the spine has no way to know.
  The headline is "accounting," never "guaranteed accounting."
- `spin budget` is **advisory and always exits 0**. A genuinely T2 task (architecture,
  adversarial review) *should* cost a lot; punishing necessary spend with a blocking exit
  would betray the routing doctrine's own rule — the cheapest tier that *verifiably* does
  the job, not the cheapest tier period. The signal is the `over_budget` flag, not a gate.

## What measurement is NOT allowed to become

- A tokenizer or pricing table in `src/` (guard-forbidden — it would make the spine
  model-aware even with no network call).
- A gate that branches on token *magnitude* (that would let cost block correct work).
- A reason to auto-downgrade a tier (the CLI records cost; it never acts on it).
